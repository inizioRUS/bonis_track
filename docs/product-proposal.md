# Product Proposal

## 1. Обоснование идеи

Внутренние знания компаний распределены по различным системам. Сотрудники тратят значительное время на поиск и ручную агрегацию информации.

Классическая RAG-архитектура:
- плохо обрабатывает длинные и составные запросы;
- может терять важные аспекты запроса;
- подвержена hallucinations.

### Гипотеза

Добавление агентной логики (декомпозиция, итеративный retrieval, проверка ответа, deepresearch, ner, использование внешних API) повысит полноту и надежность ответов.

---

## 2. Цель и метрики

### Цель проекта

Реализовать чат поиск с RAG + агентной системой, повысить качество поиска для сложных, многошаговых запросов.

### Retrieval-метрики

- Recall@k
- MAP@k
- nDCG@k

### LLM-метрики

- Factual consistency (LLM-as-judge)
- Groundedness score
- Доля hallucinations

### Продуктовые метрики

- Доля повторных уточняющих запросов
- Среднее количество шагов до релевантного ответа

### Технические метрики

- p95 latency
- Средняя стоимость запроса
- Количество агентных шагов


---

## 3. Сценарии использования

### Базовый сценарий

> "Как устроен процесс согласования релиза в X-команде?"

Агент:
- декомпозирует запрос
- выполняет несколько retrieval
- агрегирует результаты

### Составной запрос

> "Какие SLA у сервиса Y и какие были инциденты за последний квартал?"

### Edge cases

- Очень длинный запрос
- Противоречивые документы
- Частично устаревшая информация
- Неявный контекст

---

## 4. Ограничения

### Технические ограничения

- p95 latency ≤ 5–8 секунд (PoC)
- Максимум 10 агентных шага
- Ограничение размера контекста

### Операционные ограничения

- Лимит на токены
- Ограниченный бюджет inference
- Использование open-source данных для экспериментов

---

## 5. Архитектура

Модули системы (расширенная версия):

### 1. Data ingestion

- Коннекторы к внутренним источникам: wiki / Confluence, кастомные доки, Jira, Slack / Teams, базы знаний, runbooks, системы мониторинга и инцидентов. В рамках РОС выберу ограниченный скоуп данных.
- Нормализация данных: преобразование документов в единый формат.
- Извлечение метаданных: владелец документа, команда, сервис, версия, дата обновления, источник, уровень доверия.
- Дедупликация документов.
- Инкрементальные обновления индекса.

---

### 2. Preprocessing

- Очистка текста.
- Разметка структуры документа (заголовки, списки, таблицы, код).
- Извлечение ссылок между документами.

### 3. Chunking + Embedding

Используется комбинированная стратегия разбиения документов:

- структурный chunking (по разделам документа)
- semantic chunking (по смысловым блокам)

Для поиска используются :

- Hybrid retrieval (комбинация dense + sparse)

Каждый chunk получает:

- метаданные
- ссылки на сущности из графа

---

### 4. Storage layer

Система хранения включает:

- Vector database для эмбеддингов
- Текстовый индекс для BM25
- Хранилище исходных документов

Дополнительно хранятся метаданные:

- команда
- сервис
- окружение
- источник
- свежесть данных
- уровень доверия


---

### 5. Baseline RAG

Для бейзлана будет использоваться базовый RAG pipeline:

1. Retrieval документов
2. Reranking результатов
3. Передача контекста в LLM
4. Генерация ответа

Дополнительные проверки:

- наличие источников
- проверка длины и релевантности контекста

---

### 6. Agent Orchestrator (что планируется сдлеать)

Агентная система управляет сложными запросами и координирует несколько этапов поиска.

####  Complexity analyzer

Определяет:

- сложность запроса
- необходимость декомпозиции
- необходимость deepresearch
- необходимость использования внешних API(К примеру API JIRA)
- необходимость уточняющих вопросов

---

#### Query decomposition

Разбиение сложного запроса на подзадачи.

Пример:
"Какие SLA у сервиса Y и какие были инциденты за последний квартал?"

Разбивается на:

1. Найти SLA сервиса Y  
2. Найти инциденты сервиса Y за последний квартал  
3. Сформировать итоговую сводку  

---

#### Iterative multi-step retrieval

Агент может выполнять несколько циклов поиска:
retrieval → анализ → уточнение запроса → retrieval


---

#### NER-driven planning

Сначала выполняется извлечение сущностей из пользовательского запроса.

Например:
сервис: Y
период: последний квартал


После этого строится план поиска:

1. документация сервиса
2. каталог сервисов
3. система инцидентов
4. мониторинг
5. постмортемы

---

#### External API tools

Агент может обращаться к внешним системам через API.

Это используется для получения **актуальных структурированных данных**, которые могут отсутствовать в документах.

Примеры API:

- Jira API — инциденты, тикеты, изменения
- GitHub / GitLab API — релизы, pull requests, теги
- PagerDuty / Opsgenie API — инциденты и алерты
- Prometheus / Grafana API — метрики
- Elastic / Loki API — логи
- Service catalog / CMDB API — владельцы сервисов и зависимости

Результаты API приводятся к единому формату **evidence objects** и используются вместе с документами. (Для демо возьму одно из представленных API)

---

#### Deepresearch mode

Специальный режим для сложных или длинных запросов.

Включает:

- расширение запроса (синонимы, альтернативные названия сервисов)
- переход по ссылкам между документами
- поиск связанных тикетов и постмортемов
- анализ нескольких источников

Deepresearch формирует карту источников и пытается покрыть все аспекты запроса.

---

#### Verification step

Перед генерацией финального ответа выполняются проверки:

- groundedness (утверждения имеют источник)
- обнаружение противоречий между документами
- проверка свежести информации
- сравнение данных из документов и API

Если данные конфликтуют, система может:

- показать оба значения
- указать источник
- указать дату документа

---

#### Budget и Step Controller

Контролирует:

- максимальное число шагов агента (≤ 10)
- использование токенов
- latency

Если бюджет исчерпан:

- агент завершает работу
- возвращает частичный ответ
- сообщает какие аспекты не были найдены

---

### 7. Answer generation

Финальный ответ формируется на основе собранного контекста.

Ответ включает:

- структурированное объяснение
- ссылки на источники
- цитаты из документов
- уровень уверенности ответа


---

### 8. Access control и Policy

Эти функции не делегируются LLM.

Система выполняет:

- фильтрацию документов по правам пользователя
- аудит запросов
- логирование
- контроль бюджета inference

---

### 9. Evaluation pipeline

Для оценки качества системы используется evaluation pipeline.

Источники данных:

- open-source датасеты
- синтетические запросы
- golden dataset

Метрики:

**Retrieval**

- Recall@k
- MAP@k
- nDCG@k

**LLM**

- factual consistency
- groundedness
- hallucination rate

**Agent**

- coverage аспектов запроса
- количество шагов
- процент честных ответов "информация не найдена"

**Технические**

- p95 latency
- стоимость запроса
- количество agent steps

---

## 6. Data Flow

1. Пользовательский запрос
2. Agent:
   - анализ сложности
   - решение о декомпозиции
3. Retrieval (1..N шагов)
4. Агрегация контекста
5. Verification step
6. Финальный ответ

### Делегируется LLM/Agent

- Декомпозиция запроса
- Формирование уточняющих запросов
- Агрегация информации
- Проверка согласованности ответа

### Не делегируется

- Контроль доступа
- Фильтрация по правам
- Логирование
- Контроль бюджета
