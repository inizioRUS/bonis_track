# Product Proposal

## 1. Обоснование идеи

Внутренние знания компании распределены по различным системам. Сотрудники тратят значительное время на поиск и ручную агрегацию информации.

Классическая RAG-архитектура:
- плохо обрабатывает длинные и составные запросы
- может терять важные аспекты запроса
- подвержена hallucinations

### Гипотеза

Добавление агентной логики (декомпозиция, итеративный retrieval, проверка ответа) повысит полноту и надежность ответов.

---

## 2. Цель и метрики

### Цель проекта

Повысить качество корпоративного поиска для сложных запросов.

### Retrieval-метрики

- Recall@k
- MAP@k
- nDCG@k

### LLM-метрики

- Factual consistency (LLM-as-judge)
- Groundedness score
- Доля hallucinations

### Продуктовые метрики

- Доля повторных уточняющих запросов
- Среднее количество шагов до релевантного ответа

### Технические метрики

- p95 latency
- Средняя стоимость запроса
- Количество агентных шагов

---

## 3. Сценарии использования

### Базовый сценарий

> "Как устроен процесс согласования релиза в X-команде?"

Агент:
- декомпозирует запрос
- выполняет несколько retrieval
- агрегирует результаты

### Составной запрос

> "Какие SLA у сервиса Y и какие были инциденты за последний квартал?"

### Edge cases

- Очень длинный запрос
- Противоречивые документы
- Частично устаревшая информация
- Неявный контекст

---

## 4. Ограничения

### Технические ограничения

- p95 latency ≤ 5–8 секунд (PoC)
- Максимум 3–4 агентных шага
- Ограничение размера контекста

### Операционные ограничения

- Лимит на токены
- Ограниченный бюджет inference
- Использование open-source данных для экспериментов

---

## 5. Архитектура (набросок)

Модули системы:

1. Data ingestion
2. Chunking + embedding
3. Vector storage
4. Baseline RAG
5. Agent Orchestrator:
   - Query decomposition
   - Multi-step retrieval
   - Verification
6. Answer generation
7. Evaluation pipeline

---

## 6. Data Flow

1. Пользовательский запрос
2. Agent:
   - анализ сложности
   - решение о декомпозиции
3. Retrieval (1..N шагов)
4. Агрегация контекста
5. Verification step
6. Финальный ответ

### Делегируется LLM/Agent

- Декомпозиция запроса
- Формирование уточняющих запросов
- Агрегация информации
- Проверка согласованности ответа

### Не делегируется

- Контроль доступа
- Фильтрация по правам
- Логирование
- Контроль бюджета
